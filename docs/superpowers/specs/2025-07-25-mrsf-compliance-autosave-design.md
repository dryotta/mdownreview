# MRSF v1.0 Spec Compliance & Auto-Save Reliability

**Date:** 2025-07-25
**Status:** Approved
**Scope:** Fix MRSF spec compliance gaps and auto-save reliability issues

## Problem Statement

The review sidecar files generated by mdownreview do not strictly follow the MRSF v1.0 specification in several areas, and the auto-save mechanism can lose comments when switching tabs quickly.

**Spec reference:** https://github.com/wictorwilen/MRSF/blob/main/MRSF-v1.0.md

## Changes

### 1. ID Generation → UUIDv4

**Spec §6.1:** `id` SHOULD be collision-resistant (e.g., UUIDv4 or ULID).

**Current:** `Math.random().toString(36).slice(2,10)` — 8-char alphanumeric, high collision risk.

**Fix:** Replace with `crypto.randomUUID()` which produces standard UUIDv4 strings. Extract to a shared `generateCommentId()` utility in `src/lib/comment-utils.ts` for reuse and testability.

### 2. Document Field → Relative Path

**Spec §4:** `document` is the relative path from the repository or workspace root to the annotated file.

**Current:** Both viewers extract just the filename: `filePath.split(/[/\\]/).pop()`.

**Fix:** When workspace `root` is set, compute the relative path from root to file. When no root is set, fall back to the filename. Pass the `root` from Zustand store to the save call. The `useAutoSaveComments` hook (item 9) will handle this.

### 3. Delete Comment → Promote Replies (§9.1)

**Spec §9.1:** When a parent comment is removed, implementations MUST promote its direct replies:
1. Copy targeting fields (`line`, `end_line`, `start_column`, `end_column`, `selected_text`, `selected_text_hash`) from parent to reply if the reply omits them.
2. Update `reply_to` to the parent's `reply_to` (grandparent) or remove it if parent was root.
3. Remove the parent comment.

**Current:** `deleteComment` just `filter()`s out the comment, orphaning all replies.

**Fix:** In `deleteComment`, before filtering, find direct replies and:
- Copy missing targeting fields from the deleted parent
- Reparent `reply_to` to the grandparent (or remove if parent was root)

### 4. Selected Text Length Cap (4096 chars)

**Spec §6.2:** `selected_text` MUST NOT exceed 4096 characters.

**Current:** No validation.

**Fix:** Truncate `selected_text` to 4096 characters in `createSelectionAnchor()` and in the new line-comment anchor creation. Also truncate `selected_text_hash` recomputation on the truncated value.

### 5. Comment Text Length Soft Cap (16384 chars)

**Spec §6.1:** `text` SHOULD NOT exceed 16384 characters.

**Current:** No validation.

**Fix:** In `CommentInput`, enforce a 16384-character limit in the textarea with a character counter when approaching the limit (e.g., show count when >15000 chars). Disable submit when over limit.

### 6. Line Comments Include Full Line as Selected Text

**Spec §6.2 on `selected_text`:** "For line-only comments, `selected_text` SHOULD contain the full content of the referenced line (excluding the trailing newline)."

**Current:** `LineCommentMargin.handleSave` passes `{ line: lineNumber }` only — no `selected_text`.

**Fix:** In `LineCommentMargin.handleSave`, when no `onSaveComment` override is provided (i.e., it's a plain line comment, not a selection):
1. Use `lineText` prop as `selected_text`
2. Compute `selected_text_hash` via `computeSelectedTextHash(lineText)`
3. Pass the full anchor `{ line, selected_text, selected_text_hash }` to `addComment`

This is async (hash computation), so `handleSave` becomes async.

### 7. Auto-Populate `commit` Field

**Spec §6.2:** `commit` — Git commit hash associated with the comment. When absent, line/column anchors are best-effort.

**Current:** `getGitHead` command exists but is never called during comment creation.

**Fix:** In `addComment` and `addReply` (store), attempt to call `getGitHead` for the file path. Since this is async and the store actions are synchronous, the approach is:
- The `useAutoSaveComments` hook (or a new `useCommitEnricher` effect) will enrich comments that lack a `commit` field by calling `getGitHead` once per file and caching the result.
- Best-effort: if git is unavailable or the file isn't in a repo, `commit` remains omitted.

### 8. Cross-Field Validation

**Spec §7.1:** `end_line` MUST be ≥ `line`; `end_column` MUST be ≥ `start_column` when on the same line.

**Current:** No validation.

**Fix:** Add validation in `createSelectionAnchor()`:
- Clamp `end_line` to be ≥ `line`
- Clamp `end_column` to be ≥ `start_column` when `line === end_line`
- Log a warning if clamping occurs

### 9. Shared `useAutoSaveComments` Hook

**Problem:** Both MarkdownViewer and SourceView have identical auto-save `useEffect` blocks with a 500ms debounce. When a tab is switched (component unmounts), `clearTimeout` cancels the pending save, losing the comment.

**Fix:** Extract into `src/hooks/useAutoSaveComments.ts`:

```typescript
export function useAutoSaveComments(filePath: string, comments: MrsfComment[] | undefined) {
  // ...
}
```

Behavior:
- 500ms debounce on `comments` changes (same as current)
- **Flush on unmount**: in the cleanup function, if a save is pending, execute it immediately (synchronous fire-and-forget via `saveReviewComments`)
- Compute `document` as relative path from workspace `root` (from store) when available
- Log save errors via `logger.ts` instead of `.catch(() => {})`
- Track `loadedRef` internally to prevent saving before initial load completes
- Both viewers replace their inline save `useEffect` with this hook

### 10. AGENTS.md Enum Fixes

The AGENTS.md comment data model section lists incorrect enum values:
- `type: "suggestion"  # issue | suggestion | question | note | praise` → Should reference spec values: suggestion, issue, question, accuracy, style, clarity
- `severity: "warning"  # critical | warning | info | nitpick` → Should reference spec values: low, medium, high
